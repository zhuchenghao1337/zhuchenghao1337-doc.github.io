# MVC架构模式

把一个交互式应用程序划分为3个部分：

  • 模型：包含业务逻辑和数据
  • 视图：进行数据展示
  • 控制器：接受输入，选择业务逻辑实现，选择试图响应

优点：

  1. 多个视图共享一个模型，提高代码可重用行。
  2. 三个模块相互独立，松耦合架构
  3. 控制器提高了应用程序的灵活性和可配置性

缺点：

  1. 原理复杂
  2. 增加了系统结构和实现的复杂性
  3. 视图对模型数据的访问效率低

# 1.工厂模式

工厂模式通常有三种模式：

  • 简单工厂模式（Simple Factory) ：又叫静态 工厂模式，不属于23种设计模式之一
  • 工厂方法模式（Factory Method）：23种设计模式之一
  • 抽象工厂模式（Abstract Factory）：23种设计模式之一

简单工厂模式

  • 又叫静态工厂模式，不属于23种设计模式
  • 有三种角色：抽象产品、具体产品和静态工厂
  • 优点：将创建对象（生产）和使用（消费）分离，消费者不需要关心怎么生产，直接用就ok了
  • 缺点：不满足开闭原则，扩展需要修改工厂类；工厂的责任过于大。

# 代码/框架的侵入式

侵入式：

这个代码/模块嵌入了其他的代码，可能是引入的框架，或者是通过接口继承得到的。

缺点

这些代码不引入框架就没法使用，没法单元测试。

非侵入式：

比如spring框架是非侵入式的，你引入spring的相关依赖，就可以使用。

# 单例还是多例

# 概念：

单例和多例是设计模式中的两个概念。

单例模式是指一个类只能创建一个对象实例，并且该对象实例在整个程序运行期间只会被创建一次。这种模式的使用可以有效地节省系统资源，避免了创建多个对象实例带来的性能开销。在实际开发中，单例模式常常被用来管理全局变量、日志输出、数据库连接等场景。

多例模式是指一个类可以创建多个对象实例，并且这些实例之间相互独立。每个对象实例可以有不同的属性或状态，而且可以根据需求动态地创建或销毁。多例模式在实际开发中应用较少，但在某些场景下，例如线程池、数据库连接池等情况下，可以有效地管理资源和提高程序性能。

# spring的单例和多例

spring在默认情况下，在根据配置文件new Spring上下文对象时，同时创建被管理的bean对象，这些bean对象是单例的。

可以在配置文件中，配置scope属性为prototype（默认为singleton），此时new 上下文对象时不会创建该对象，只有当调用getBean方法获取对象时才会创建对象，而且这个bean对象时多例的（每调用一次getBean方法就会新new一个对象）。
